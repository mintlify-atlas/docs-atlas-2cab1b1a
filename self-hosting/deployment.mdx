The Svix server is a single Rust binary. This page covers the three most common ways to get it running: Docker Compose (easiest), a standalone Docker container, and building from source.

## Runtime dependencies

The server requires:

- **PostgreSQL** — for storing events. Only PostgreSQL is currently supported.
- **Redis 6.2.0 or higher** (optional) — for the task queue and cache. Strongly recommended in production.

<Warning>
If you use Redis, enable persistence so the task queue survives restarts. Set `maxmemory-policy` to `noeviction` or a `volatile-*` policy. Do not use an eviction policy that can remove keys without an explicit TTL.
</Warning>

## Option 1: Docker Compose

Docker Compose is the quickest way to get a complete stack running. The included `docker-compose.yml` boots PostgreSQL, PgBouncer, Redis, and the Svix server automatically.

<Steps>

<Step title="Clone the repository">

```bash
git clone https://github.com/svix/svix-webhooks
cd svix-webhooks/server
```

</Step>

<Step title="Set a JWT secret">

Open `docker-compose.yml` and uncomment the `SVIX_JWT_SECRET` line, replacing `x` with a securely generated random string:

```yaml docker-compose.yml
environment:
  WAIT_FOR: "true"
  SVIX_REDIS_DSN: "redis://redis:6379"
  SVIX_DB_DSN: "postgresql://postgres:postgres@pgbouncer/postgres"
  SVIX_JWT_SECRET: "your-secret-here"  # set a real secret
```

<Warning>
Never run in production without setting a real `SVIX_JWT_SECRET`. The JWT secret cannot be changed after first use.
</Warning>

</Step>

<Step title="Start the stack">

```bash
docker compose up
```

The server listens on port `8071`. The compose file includes the following services:

| Service      | Image                         | Purpose                        |
| ------------ | ----------------------------- | ------------------------------ |
| `backend`    | `svix/svix-server`            | Svix API and message worker    |
| `postgres`   | `postgres:13.4`               | Primary data store             |
| `pgbouncer`  | `edoburu/pgbouncer:1.15.0`    | PostgreSQL connection pooler   |
| `redis`      | `redis:7-alpine`              | Task queue and cache           |

Redis is configured with AOF persistence (`--appendonly yes --appendfsync everysec`) and saves a snapshot every 60 seconds if at least 500 keys have changed.

</Step>

</Steps>

## Option 2: Standalone Docker container

Run a standalone container when you already have PostgreSQL and Redis available. Pass configuration via environment variables using a `.env` file or individual `-e` flags.

<Steps>

<Step title="Create an environment file">

Create a file named `svix.env` with the required configuration:

```bash svix.env
SVIX_JWT_SECRET="your-secret-here"
SVIX_DB_DSN="postgresql://postgres:postgres@your-postgres-host/postgres"
SVIX_REDIS_DSN="redis://your-redis-host:6379"
SVIX_QUEUE_TYPE="redis"
SVIX_CACHE_TYPE="memory"
```

</Step>

<Step title="Run the container">

```bash
docker run \
  --name svix-server \
  -p 8071:8071 \
  --env-file svix.env \
  svix/svix-server
```

You can use a specific version tag instead of the default `latest`:

```bash
docker run \
  --name svix-server \
  -p 8071:8071 \
  --env-file svix.env \
  svix/svix-server:1.90.0
```

</Step>

</Steps>

## Option 3: Building from source

The Svix server is written in Rust. You need a working Rust toolchain to build it.

<Steps>

<Step title="Install Rust">

Install `rustup` and set the stable toolchain:

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
rustup default stable
```

</Step>

<Step title="Clone and build">

```bash
git clone https://github.com/svix/svix-webhooks
cd svix-webhooks/server
cargo install --path svix-server
```

This installs the `svix-server` binary into your Cargo bin directory (typically `~/.cargo/bin`).

</Step>

<Step title="Generate a JWT and run">

Before starting the server, generate an authentication token:

```bash
svix-server jwt generate
```

Then start the server with your configuration in place (see [configuration](#configuration) below):

```bash
svix-server
```

</Step>

</Steps>

## Configuration

The server reads configuration from three sources, in order of precedence:

1. Environment variables prefixed with `SVIX_`
2. A `.env` file in the working directory
3. A `config.toml` file in the working directory

### Key options

| Config key    | Env variable        | Description                                                    |
| ------------- | ------------------- | -------------------------------------------------------------- |
| `jwt_secret`  | `SVIX_JWT_SECRET`   | Secret used to sign and verify authentication tokens. Required. Cannot be changed after first use. |
| `db_dsn`      | `SVIX_DB_DSN`       | PostgreSQL connection string. Only PostgreSQL is supported.    |
| `redis_dsn`   | `SVIX_REDIS_DSN`    | Redis connection string. Leave unset to disable Redis.         |
| `queue_type`  | `SVIX_QUEUE_TYPE`   | Message queue backend: `memory`, `redis`, or `rediscluster`.   |
| `cache_type`  | `SVIX_CACHE_TYPE`   | Cache backend: `memory`, `redis`, `rediscluster`, or `none`.   |
| `listen_address` | `SVIX_LISTEN_ADDRESS` | Address and port the server binds to. Default: `0.0.0.0:8071`. |

### Example `config.toml`

```toml config.toml
# Required: secret used to sign JWTs. Must not be changed after first run.
jwt_secret = "8KjzRXrKkd9YFcNyqLSIY8JwiaCeRc6WK4UkMnSW"

# Required: PostgreSQL DSN
db_dsn = "postgresql://postgres:postgres@pgbouncer/postgres"

# Optional: Redis DSN (required when queue_type or cache_type is "redis")
redis_dsn = "redis://redis:6379"

# Message queue backend
queue_type = "redis"

# Cache backend
cache_type = "memory"
```

### Example environment file

```bash development.env
SVIX_JWT_SECRET=your-secret-here
SVIX_DB_DSN=postgresql://postgres:postgres@localhost/postgres
SVIX_REDIS_DSN=redis://localhost
SVIX_QUEUE_TYPE=redis
SVIX_CACHE_TYPE=memory
SVIX_LOG_LEVEL=info
```

<Tip>
Use `queue_type = "memory"` during local development if you don't have Redis available. Switch to `"redis"` in production to persist the queue across restarts.
</Tip>

## Generating an API token

Once the server is running, generate a JWT to use as your API token:

```bash
svix-server jwt generate
```

Use the output as a `Bearer` token in the `Authorization` header when calling the API, or pass it directly to the Svix SDK:

```bash
Authorization: Bearer <token>
```

See the [Quickstart](/quickstart) for how to initialize the SDK with this token.
